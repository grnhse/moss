Thors method name limitations

May 13, 2016 • Juan C. Muller

We designed a RESTy api consumer not too long ago, and we wanted to use resource actions. Let’s say, users.

We conceived the following API:

$ my_app users                                # Would list users
$ my_app users:delete [ID]                    # Would delete user with id ID
$ my_app users:new [NAME] [EMAIL] [ATTRIBUTE] # Would create an user

… and we wanted to use Thor.

In thor, you define a command line program very easily. It takes care of argument parsing, option checking, defaults setting, etc. For example:

# === blah.rb ===
require 'thor'

class Blah < Thor
  def hello(arg)
    puts "Hello, #{arg}"
  end
end

Blah.start
$ blah.rb hello world
# Hello, world

Since your program actions map 1:1 to methods in your thor class, you can’t define anything named users:delete, since it is a syntax error.

What to do?

Luckily, one of Ruby’s greatest strengths is its metaprogramming. What is it? Well, that’s a subject worthy of a few books… not a few lines :)

For now, let’s just focus on define_method. You can call define_method on any object to create a new method at runtime. 

> a = Object.new
=> #<Object:0x007f8ce51cf920>
> a.respond_to?(:foo)
=> false
> a.class.send(:define_method, :foo) do  "bar" end
=> :foo
> a.respond_to?(:foo)
=> true
> a.foo
=> "bar"

It’s syntax is a bit cumbersome when you’re modifying an already instantiated object, of course. But, at class definition time, it allows you to declare methods dynamically. For example:

class Foo
  %w(a b c).each do |method_name|
    define_method(method_name) do |*params|
      puts "#{self.class.name}##{method_name}(#{params.join(", ")})"
    end
  end
end
> foo = Foo.new
=> #<Foo:0x007f9d1c942fa8>
> foo.a
Foo#a()
=> nil
> foo.b "a", "B"
Foo#b(a, B)
=> nil
> foo.c "a", "B"
Foo#c(a, B)
=> nil
> foo.c [:a]
Foo#c(a)

So, putting it all together:

require 'thor'

class MyCli < Thor
  desc "users:new [NAME] [EMAIL] [ATTRIBUTE]", "Create an user named [NAME]..."
  define_method("users:new") do |name, email, attribute|
    # ... do something here that creates
    # the user and returns something
    # meaningful...
  end
end
$ my_cli help
Commands:
  my_cli help [COMMAND]                        # Describe available commands or one specific command
  my_cli users:new [NAME] [EMAIL] [ATTRIBUTE]  # Create an user named [NAME]...

Cheers!
